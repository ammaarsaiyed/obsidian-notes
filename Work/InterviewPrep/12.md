## 4. Technical Themes To Land

### Designing AI Solutions

Always start with the workflow, not the model:

- What decision or process are we improving?
- Who uses the output?
- What happens if it is wrong?
- What data is used?
- What is success?
- What risk level are we operating at?

Functional requirements:

- inputs, outputs, tools, integrations, user journey, structured response, escalation.

Non-functional requirements:

- latency, reliability, availability, cost, security, explainability, auditability, scalability, maintainability, monitoring.

Approach selection:

- deterministic workflow if rules are stable
- traditional ML if structured prediction is needed
- RAG if internal/current knowledge is needed
- tool use if live systems/structured data are needed
- agentic workflow if multi-step reasoning and dynamic tool selection are genuinely needed

Line to land:

> A lead engineer should choose the simplest pattern that meets the business need safely.

### Agentic AI

Use agents when:

- task needs multi-step investigation
- tool selection is dynamic
- context must be gathered from multiple sources
- reasoning and state matter

Avoid agents when:

- deterministic rules are enough
- evaluation is not possible
- the workflow is stable and high-risk
- complexity outweighs value

Agent controls:

- scoped tools
- least-privilege access
- structured outputs
- source citations
- human review
- prompt-injection tests
- audit logs
- eval suite
- fallback behaviour

### RAG

Use RAG when the model needs approved internal or current knowledge. Key concerns:

- chunking
- metadata
- retrieval quality
- access control
- freshness
- citation accuracy
- latency
- unsupported claims

Evaluate RAG with:

- retrieval relevance
- answer correctness
- groundedness
- citation accuracy
- hallucination/unsupported claim rate
- user usefulness

### Evaluation

For traditional ML:

- baseline
- train/test split or time-based validation
- F1/precision/recall/calibration
- confusion matrix
- subgroup performance where relevant
- drift after release

For GenAI/agents:

- groundedness
- tool-call correctness
- retrieval relevance
- output schema validity
- safety
- prompt injection
- latency
- token cost
- human/SME review
- business usefulness

If no labels:

- SME review
- rubrics
- historical cases
- synthetic edge cases
- pairwise comparisons
- human feedback

Line to land:

> No perfect labels does not mean no evaluation. It means using multiple forms of evidence.

### Responsible AI And Governance

Risk categories:

- hallucination
- bias/unfairness
- data leakage
- prompt injection
- overreliance
- poor explainability
- weak auditability
- operational failure
- customer harm

Controls:

- use-case risk assessment
- least-privilege access
- approved data sources
- human-in-the-loop
- structured outputs
- source grounding
- model/prompt/tool versioning
- eval gates
- audit logs
- monitoring
- rollback criteria

Line to land:

> I treat AI risk as a design input, not a final approval hurdle.

### Production And MLOps

Production readiness means:

- source control and code review
- automated tests
- CI/CD
- environment separation
- infrastructure as code
- versioned prompts/config/models/tools
- monitoring dashboards
- runbooks
- rollback
- incident ownership
- handoff documentation

Monitor:

- latency, throughput, errors, timeout rate
- token usage and cost
- data freshness and drift
- prediction/output distribution
- safety flags
- retrieval/tool failures
- user feedback and business outcomes

## 5. System Design Outline

If asked to design an agentic complaints analysis MVP:

1. Define colleague workflow and decision being supported.
2. Retrieve authorised customer/transaction context.
3. Retrieve relevant policy/process guidance using RAG.
4. Use scoped tools through MCP or similar interface.
5. Produce structured output: facts, sources, possible issues, uncertainty, next checks, escalation flags.
6. Keep colleague as final decision-maker.
7. Add controls: access, audit, validation, human review.
8. Evaluate with historical cases, SME review, tool-call accuracy, groundedness, hallucination checks, latency and cost.
9. Monitor in pilot.
10. Handoff with architecture notes, eval report, known limitations, runbook and backlog.

Strong phrase:

> The MVP should support complaint handlers, not replace them. It should reduce cognitive load while keeping evidence, uncertainty and accountability visible.

> Make good engineering easier through lightweight patterns, examples, checklists and reusable modules.

- reducing friction through reusable patterns

### Inclusive

Talk about:

- sharing context before meetings
- inviting written input
- making decision criteria explicit
- seeking engineering, business, risk, architecture and user perspectives

Key phrase:

> In AI, diverse perspectives are a risk control because different people spot different harms and edge cases.

Use:

- EDP Consumer Data Products as long-term reusable assets
- reusable AI patterns
- token tracking for cost visibility
- documentation, ownership and monitoring

Key phrase:

> Sustainable engineering means avoiding short-term demos that create long-term support, governance or cost problems.

### Trust

Talk about:

- admitting gaps early
- blameless learning
- surfacing risk
- giving people ownership with support
- auditability and transparency

Key phrase:

> Hidden mistakes become incidents or rework. Visible mistakes become better controls.

### Integrity

Talk about:

- customer trust
- not overstating AI capability
- speaking up if evaluation is weak
- raising concerns with evidence and proposed mitigation

Key phrase:

> In AI, integrity means being honest about limitations because confident wrong answers can still cause real harm.
